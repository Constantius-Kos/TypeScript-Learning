## 3. 🤔 Проблема в JavaScript

В JS функцию можно вызвать с меньшим количеством аргументов, чем она ожидает — и ничего не произойдёт (пока не случится баг):

```javascript
// JavaScript
function createUser(name, role, age) {
  console.log(`${name}, роль: ${role}, возраст: ${age}`);
}

createUser("Алекс");
// Выведет: "Алекс, роль: undefined, возраст: undefined"
// Никаких ошибок! Просто тихий баг.
```

Разработчики в JS обычно защищались так:

```javascript
function createUser(name, role, age) {
  role = role || "user";       // если не передали — ставим "user"
  age = age !== undefined ? age : 18; // защита через проверку
}
```

Это работает, но выглядит как самодельный костыль. TypeScript даёт нам встроенные инструменты для этого.

---

## 4. 📖 Теория

### Optional параметры (`?`)

Ставишь `?` после имени параметра — и он становится необязательным. TypeScript будет знать, что его могут не передать.

```typescript
function greet(name: string, greeting?: string): void {
  //                                       ↑
  //               знак вопроса = "этот параметр необязательный"
}
```

**Важно:** внутри функции тип такого параметра будет `string | undefined`. Это потому, что если его не передали — там `undefined`. TypeScript честно говорит тебе об этом.

**Правило:** обязательные параметры всегда идут ПЕРВЫМИ, optional — в конце. Иначе TS выдаст ошибку.

```typescript
// ✅ Правильно
function fn(required: string, optional?: number): void {}

// ❌ Неправильно — обязательный после необязательного
function fn(optional?: number, required: string): void {}
```

---

### Default параметры (`= значение`)

Другой способ — задать значение по умолчанию прямо в сигнатуре:

```typescript
function greet(name: string, greeting: string = "Привет"): void {
  //                                              ↑
  //                          если не передали — используется "Привет"
}
```

**Ключевое отличие от `?`:**
- `greeting?: string` → тип внутри функции: `string | undefined` (нужно проверять!)
- `greeting: string = "Привет"` → тип внутри функции: всегда `string` (уже безопасно!)

**Почему default лучше в большинстве случаев?** Потому что ты сразу получаешь готовое значение и не обязан писать `if (greeting !== undefined)` внутри тела функции.

---

### Когда что использовать?

| Ситуация | Что использовать |
|---|---|
| Параметр может не передаваться, и это нормально — ты обработаешь `undefined` | `?` (optional) |
| Параметр может не передаваться, но у него есть разумное значение по умолчанию | `= value` (default) |
| Параметр всегда обязателен | ничего — обычный параметр |

## 5. 💡 Примеры с детальными комментариями

### Пример 1: Optional параметр

```typescript
// greeting? означает: можешь передать, можешь не передавать
function sayHello(name: string, greeting?: string): string {
  // Внутри greeting имеет тип: string | undefined
  // Поэтому нам нужна проверка перед использованием:
  if (greeting !== undefined) {
    return `${greeting}, ${name}!`;
  }
  return `Привет, ${name}!`; // fallback если greeting не передали
}

sayHello("Алекс");            // "Привет, Алекс!"
sayHello("Алекс", "Добрый день"); // "Добрый день, Алекс!"
```

### Пример 2: Default параметр

```typescript
// Если currency не передали — будет "RUB"
function formatPrice(price: number, currency: string = "RUB"): string {
  // currency здесь всегда string — никакого undefined!
  // Не нужно проверять — TypeScript гарантирует значение
  return `${price} ${currency}`;
}

formatPrice(1000);         // "1000 RUB"
formatPrice(50, "USD");    // "50 USD"
```

### Пример 3: Реальный use case — создание пользователя

```typescript
type UserRole = "admin" | "editor" | "viewer";

type User = {
  name: string;
  role: UserRole;
  age: number;
};

// role по умолчанию "viewer", age — необязательный
function createUser(name: string, role: UserRole = "viewer", age?: number): User {
  return {
    name,
    role,
    age: age ?? 0, // если age не передали (undefined), используем 0
    //      ↑↑
    //  nullish coalescing — помнишь из Урока 6?
  };
}

const user1 = createUser("Алекс");              // role="viewer", age=0
const user2 = createUser("Мария", "admin");     // role="admin", age=0
const user3 = createUser("Влад", "editor", 25); // role="editor", age=25
```

## 6. 🔍 JS vs TS сравнение

**Было в JS (самодельная защита):**
```javascript
function sendEmail(to, subject, body, cc) {
  cc = cc || [];           // Костыль: если не передали — пустой массив
  subject = subject || "Без темы"; // ещё один костыль
  // ...
}
```

**Стало в TS (чисто и явно):**
```typescript
function sendEmail(
  to: string,
  subject: string = "Без темы",
  body: string,
  cc: string[] = []
): void {
  // Всё чисто, никаких костылей. TypeScript сам следит.
}
```

Преимущества:
- Сразу видно какие параметры необязательны прямо из сигнатуры
- Нет болезненного `|| []` который ломается на `0` или `false`
- IDE автодополняет и подсказывает значения по умолчанию

## 9. 🛠️ Практический совет

В реальных проектах **default параметры встречаются намного чаще** чем optional. Причина простая: если параметр необязателен — обычно уже есть разумное значение по умолчанию. Писать `= значение` чище и безопаснее, чем потом проверять `if (param !== undefined)` внутри функции.

Optional `?` чаще используют когда параметр действительно факультативный и у него нет никакого "разумного умолчания" — например, коллбек-функция или дополнительные настройки.
