# Нотатки — Урок 7: Type Narrowing

## ❓ Питання і відповіді з сесії

### Q: Exhaustiveness check через never — часто використовується на практиці?
Так, в конкретних сценаріях:
- **Redux reducers** — обробка всіх action types
- **State machines** — обробка всіх станів
- **API responses** — обробка всіх варіантів відповіді

Цінність: при додаванні нового варіанту в union — компілятор сам знайде всі місця, де потрібно додати обробку. Працює як "список завдань від компілятора".

---

### Q: Навіщо функція-хелпер assertNever? В чому сенс?
Замість двох рядків щоразу:
```ts
const _: never = notification;
throw new Error('Unknown');
```
Пишеш один:
```ts
return assertNever(notification);
```
Всередині функції — той самий `throw new Error`, просто запакований. Чистіше і не повторюєшся.

---

### Q: Як спрацьовує викид помилки якщо приходить never?
Два різних механізми на двох етапах:
- **Розробка (compile time):** TypeScript не дає скомпілювати код якщо тип не `never` → в продакшн взагалі не потрапить
- **Продакшн (runtime):** TypeScript вже немає, є тільки JS. Якщо з бекенду прийшов неочікуваний тип — `throw new Error` всередині функції спіймає це і впаде з зрозумілим повідомленням

---

### Q: `never` у функції означає що вона зависає в циклі?
Ні. `never` — це опис того, що функція **ніколи не поверне значення**. Два способи:
- Завжди кидає помилку (`throw`) — наш випадок
- Нескінченний цикл (`while(true){}`) — інший випадок

В `assertNever` функція завершується миттєво через `throw`, просто без `return`.

---

### Q: Навіщо `JSON.stringify(value)` всередині assertNever?
Для читабельного повідомлення про помилку:
```ts
// Без JSON.stringify:
"Необроблений випадок: [object Object]"  // ← марно

// З JSON.stringify:
"Необроблений випадок: {"kind":"slack","channel":"#general"}"  // ← видно що прийшло
```
`JSON.stringify` перетворює будь-який об'єкт на читабельний рядок.

---

### Q: Чи можна не вказувати `: never` в типі, що повертається?
В більшості випадків TS сам виведе `never`. Але явно вказують:
1. Для документації — одразу видно що функція навмисно не повертає
2. Захист: якщо хтось випадково додасть `return` — TS одразу покаже помилку

Для `assertNever` важливіший **параметр** `value: never`, а не тип, що повертається.

---

### Q: Як працює `value: never` в параметрі?
`never` в параметрі — це замок. TS не дозволить передати туди нічого, крім значення з типом `never`.

TS викреслює оброблені кейси в switch:
- Всі кейси оброблені → залишилося `never` → передати в `assertNever` можна ✅
- Забули кейс → залишилося `SlackNotification` → передати в `assertNever` не можна ❌ → помилка компіляції

---

### Q: never — для розробки, throw — для рантайму. Вірно?
Так, саме! Розділена логіка:
- `value: never` — перевірка на етапі розробки (TypeScript блокує компіляцію)
- `throw new Error(...)` — страховка на рантаймі (якщо дані прийшли ззовні і обійшли TS)
