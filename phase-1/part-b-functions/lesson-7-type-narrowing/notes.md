# Заметки — Урок 7: Type Narrowing

## ❓ Вопросы и ответы из сессии

### Q: Exhaustiveness check через never — часто используется на практике?
Да, в конкретных сценариях:
- **Redux reducers** — обработка всех action types
- **State machines** — обработка всех состояний
- **API responses** — обработка всех вариантов ответа

Ценность: при добавлении нового варианта в union — компилятор сам найдёт все места, где нужно добавить обработку. Работает как "список задач от компилятора".

---

### Q: Зачем функция-хелпер assertNever? В чём смысл?
Вместо двух строк каждый раз:
```ts
const _: never = notification;
throw new Error('Unknown');
```
Пишешь одну:
```ts
return assertNever(notification);
```
Внутри функции — тот же `throw new Error`, просто упакован. Чище и не повторяешься.

---

### Q: Как срабатывает выброс ошибки если приходит never?
Два разных механизма на двух этапах:
- **Разработка (compile time):** TypeScript не даёт скомпилировать код если тип не `never` → в продакшн вообще не попадёт
- **Продакшн (runtime):** TypeScript уже нет, есть только JS. Если с бэкенда пришёл неожиданный тип — `throw new Error` внутри функции поймает это и упадёт с понятным сообщением

---

### Q: `never` у функции означает что она зависает в цикле?
Нет. `never` — это описание того, что функция **никогда не вернёт значение**. Два способа:
- Всегда бросает ошибку (`throw`) — наш случай
- Бесконечный цикл (`while(true){}`) — другой случай

В `assertNever` функция завершается мгновенно через `throw`, просто без `return`.

---

### Q: Зачем `JSON.stringify(value)` внутри assertNever?
Для читаемого сообщения об ошибке:
```ts
// Без JSON.stringify:
"Необработанный случай: [object Object]"  // ← бесполезно

// С JSON.stringify:
"Необработанный случай: {"kind":"slack","channel":"#general"}"  // ← видно что пришло
```
`JSON.stringify` превращает любой объект в читаемую строку.

---

### Q: Можно ли не указывать `: never` в возвращаемом типе?
В большинстве случаев TS сам выведет `never`. Но явно указывают:
1. Для документации — сразу видно что функция намеренно не возвращает
2. Защита: если кто-то случайно добавит `return` — TS сразу покажет ошибку

Для `assertNever` важнее **параметр** `value: never`, а не возвращаемый тип.

---

### Q: Как работает `value: never` в параметре?
`never` в параметре — это замок. TS не позволит передать туда ничего, кроме значения с типом `never`.

TS вычёркивает обработанные кейсы в switch:
- Все кейсы обработаны → осталось `never` → передать в `assertNever` можно ✅
- Забыли кейс → осталось `SlackNotification` → передать в `assertNever` нельзя ❌ → ошибка компиляции

---

### Q: never — для разработки, throw — для рантайма. Верно?
Да, именно! Разделённая логика:
- `value: never` — проверка на этапе разработки (TypeScript блокирует компиляцию)
- `throw new Error(...)` — страховка на рантайме (если данные пришли извне и обошли TS)
