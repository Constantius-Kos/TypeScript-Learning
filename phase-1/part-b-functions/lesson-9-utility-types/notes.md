# Мої нотатки — Урок 9: Utility Types

## DTO (Data Transfer Object)

**DTO** — патерн проектування. Для кожної операції створюється окремий тип з рівно тими даними, які потрібні.

**Навіщо:** у `User` багато полів, але різним операціям потрібні різні підмножини:

| Операція | DTO |
|---|---|
| Реєстрація | `CreateUserDto` — name, email, password |
| Вхід | `LoginDto` — email, password |
| Оновлення | `UpdateUserDto` — тільки поля що змінюються |
| Відповідь API (список) | `UserPreviewDto` — id, name |
| Відповідь API (профіль) | `UserResponseDto` — все крім password |

**Принципи:**
1. Один DTO — одна операція
2. DTO — це межа системи (дані входять / виходять)
3. Всередині системи — повна модель, на межах — DTO

**DTO — це не завжди "шматочок"**
DTO необов'язково має менше полів. Це адаптоване представлення об'єкта для конкретного завдання:
- Іноді менше полів (`Pick`, `Omit`)
- Іноді ті самі поля, але з іншими правилами (`Partial` — все опціонально)
- Іноді і те й інше (`Partial<Omit<User, "id">>`)

**Не відправляй на фронт весь об'єкт — використовуй DTO**
Відправляти повного `Hero` на фронт коли потрібні тільки кілька полів — погана практика:
- **Безпека** — `passwordHash`, `adminToken` та внутрішні прапорці видні в DevTools
- **Продуктивність** — зайві кілобайти на кожен запит, боляче на мобільному
- **Зв'язність** — фронт залежить від полів які йому не потрібні, складніше змінювати структуру

Рішення:
```ts
type FrontendHeroDto = Pick<Hero, "id" | "name" | "level" | "inventory" | "hp">;
```

**DTO не перезаписує основний об'єкт автоматично**
DTO — це конверт для доставки даних, а не сам об'єкт у базі. Типовий флоу:
```
Клієнт → надсилає UpdateInventoryDto
    ↓
Сервер → дістає User з бази
    ↓
Сервер → застосовує зміни: user.inventory = dto.inventory
    ↓
Сервер → зберігає повний User назад у базу
    ↓
Клієнт ← отримує UserResponseDto (без пароля тощо)
```
DTO прийшов, дані взяли, застосували до реального об'єкта, зберегли об'єкт. DTO виконав свою роботу і забутий.

**З Utility Types:**
```ts
type CreateUserDto   = Omit<User, "id" | "createdAt">;
type UpdateUserDto   = Partial<Omit<User, "id">>;
type UserResponseDto = Omit<User, "password">;
type UserPreviewDto  = Pick<User, "id" | "name">;
```
