# Мої нотатки — Урок 9: Utility Types

## DTO (Data Transfer Object)

**DTO** — паттерн проектирования. Для каждой операции создаётся отдельный тип с ровно теми данными, которые нужны.

**Зачем:** у `User` много полей, но разным операциям нужны разные подмножества:

| Операция | DTO |
|---|---|
| Регистрация | `CreateUserDto` — name, email, password |
| Вход | `LoginDto` — email, password |
| Обновление | `UpdateUserDto` — только изменяемые поля |
| Ответ API (список) | `UserPreviewDto` — id, name |
| Ответ API (профиль) | `UserResponseDto` — всё кроме password |

**Принципы:**
1. Один DTO — одна операция
2. DTO — это граница системы (данные входят / выходят)
3. Внутри системы — полная модель, на границах — DTO

**DTO — это не всегда "кусочек"**
DTO необязательно меньше полей. Это адаптированное представление объекта для конкретной задачи:
- Иногда меньше полей (`Pick`, `Omit`)
- Иногда те же поля, но с другими правилами (`Partial` — всё опционально)
- Иногда и то и другое (`Partial<Omit<User, "id">>`)

**Не отправляй на фронт весь объект — используй DTO**
Отправлять полного `Hero` на фронт когда нужны только несколько полей — плохая практика:
- **Безопасность** — `passwordHash`, `adminToken` и внутренние флаги видны в DevTools
- **Производительность** — лишние килобайты на каждый запрос, больно на мобильном
- **Связность** — фронт зависит от полей которые ему не нужны, сложнее менять структуру

Решение:
```ts
type FrontendHeroDto = Pick<Hero, "id" | "name" | "level" | "inventory" | "hp">;
```

**DTO не перезаписывает основной объект автоматически**
DTO — это конверт для доставки данных, а не сам объект в базе. Типичный флоу:
```
Клиент → присылает UpdateInventoryDto
    ↓
Сервер → достаёт User из базы
    ↓
Сервер → применяет изменения: user.inventory = dto.inventory
    ↓
Сервер → сохраняет полный User обратно в базу
    ↓
Клиент ← получает UserResponseDto (без пароля и т.д.)
```
DTO пришёл, данные взяли, применили к реальному объекту, сохранили объект. DTO свою работу сделал и забыт.

**С Utility Types:**
```ts
type CreateUserDto   = Omit<User, "id" | "createdAt">;
type UpdateUserDto   = Partial<Omit<User, "id">>;
type UserResponseDto = Omit<User, "password">;
type UserPreviewDto  = Pick<User, "id" | "name">;
```
