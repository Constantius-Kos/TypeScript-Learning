# 1. 📌 Тема уроку: Function types і type signatures (Типізація функцій)

## 2. 🎯 Ціль
Після цього уроку ти навчишся суворо описувати, які аргументи приймає функція і що саме вона повинна повертати. Більше жодних сюрпризів у вигляді `undefined` або `NaN` через невірно передані дані!

## 3. 🤔 Проблема в JavaScript
Згадаємо класичний біль JavaScript: у функцію можна передати *що завгодно*, і вона поверне *що завгодно* (або взагалі зламається в процесі).

```javascript
// JavaScript
function calculateDiscount(price, discountPercentage) {
  return price - (price * (discountPercentage / 100));
}

// Проблема 1: Забули другий аргумент (повернеться NaN)
calculateDiscount(100);

// Проблема 2: Передали не той тип (повернеться NaN)
calculateDiscount("100$", 20);

// Проблема 3: Переплутали місцями аргументи
calculateDiscount(20, 100);
```
В JS ми дізнаємося про ці помилки тільки коли код впаде у браузері у користувача. Ось чому нам потрібен TypeScript — щоб ловити це прямо в редакторі!

## 4. 📖 Теорія

Щоб типізувати функцію в TypeScript, нам потрібно описати дві речі:
1. **Параметри (що входить)** — типізуються так само, як звичайні змінні.
2. **Тип значення, що повертається (що виходить, ReturnType)** — пишеться після дужки параметрів через двокрапку.

**Синтаксис (Type Signature / Сигнатура типу):**
```typescript
function іміяФункції(параметр1: Тип1, параметр2: Тип2): ТипПовернення {
  // код
}
```

**Якщо функція нічого не повертає:**
Іноді функції потрібні просто щоб щось зробити (вивести в консоль, змінити зовнішній стейт), але вони не роблять `return`. В JS такі функції неявно повертають `undefined`. У TypeScript для цього є спеціальний тип `void` (порожнеча).

*Зв'язок з JS:* В рантаймі вся ця типізація зникне, залишиться звичайний JS код. Типізація функцій — це security-контроль на вході до клубу твоєї функції.

## 5. 💡 Приклади з детальними коментарями

### Приклад 1: Проста функція (hello world рівень)
```typescript
// Приймає рядок, нічого не повертає (void)
function greetUser(name: string): void {
  console.log(`Привіт, ${name}!`);
  // return тут викличе помилку TS, оскільки ми вказали void
}

greetUser("Алекс"); // Працює чудово
// greetUser(42);   // Помилка TS: аргумент типу 'number' не можна призначити параметру типу 'string'
```

### Приклад 2: Трохи складніше (повертає значення)
```typescript
// Приймає два числа, повертає число
function multiply(x: number, y: number): number {
  return x * y; // Якщо спробувати повернути рядок, TS видасть помилку!
}

const result = multiply(5, 10); // TS розуміє, що result — це number
```

### Приклад 3: Реальний use case (типи-об'єкти в аргументах)
Часто ми передаємо у функцію цілий об'єкт. Найкраще описати його через `type` (Type Alias, який ми пройшли в Уроці 8!).
```typescript
type User = {
  id: string;
  name: string;
  age: number;
}

// Функція приймає об'єкт типу User і повертає boolean
function isAdult(user: User): boolean {
  return user.age >= 18;
}

const boy: User = { id: "1", name: "Влад", age: 16 };
const canBuyAlcohol = isAdult(boy); // false
```

## 6. 🔍 JS vs TS порівняння

**Було в JS:**
```javascript
const formatPrice = (price, currency) => {
  return `${price} ${currency}`;
};
formatPrice(100); // Поверне "100 undefined" (помилка через неуважність)
```

**Стало в TS:**
```typescript
const formatPrice = (price: number, currency: string): string => {
  return `${price} ${currency}`;
};
// formatPrice(100); // TS не дасть зібрати код: "Expected 2 arguments, but got 1."
```
*Перевага:* Помилки ловляться під час написання коду. Редактор сам підказує і каже: "Ей, ти забув передати валюту!". Хіба не диво?

3. НАВІЩО ми типізуємо функції, яку головну проблему ми вирішуємо порівняно з JS?

## 9. 🛠️ Практична порада
Завжди намагайся **явно** вказувати тип, що повертається у функції (`: Type { ... }`), навіть якщо TypeScript може здогадатися (type inference) сам по `return`.
Чому? Якщо всередині складної функції ти випадково зробиш `return` не того типу даних, TS без явного вказання типу просто скаже "ну гаразд, тепер функція повертає ЦЕ". Явне вказання — це твоя страховка від самого себе.

## 📝 Додаткові нотатки

**Питання:** Чому при створенні двох файлів (`practice.ts` і `solution.ts`) з однаковими іменами функцій/типів TypeScript лається на "Duplicate function implementation / Duplicate identifier"?
**Відповідь:** За замовчуванням TypeScript вважає всі файли в проекті, в яких НЕМАЄ `import` або `export`, **глобальними скриптами** (вони всі знаходяться в одній глобальній області видимості). Тому TS бачить два `Product` в одному проекті і б'є на сполох.
Щоб вирішити це, достатньо додати `export {};` в кінець файлу. Це перетворює файл на ізольований **модуль**, і імена перестають конфліктувати!
