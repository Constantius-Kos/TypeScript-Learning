# 1. 📌 Тема урока: Function types и type signatures (Типизация функций)

## 2. 🎯 Цель
После этого урока ты научишься строго описывать, какие аргументы принимает функция и что именно она должна возвращать. Больше никаких сюрпризов в виде `undefined` или `NaN` из-за неверно переданных данных!

## 3. 🤔 Проблема в JavaScript
Вспомним классическую боль JavaScript: в функцию можно передать *что угодно*, и она вернёт *что угодно* (или вообще сломается в процессе).

```javascript
// JavaScript
function calculateDiscount(price, discountPercentage) {
  return price - (price * (discountPercentage / 100));
}

// Проблема 1: Забыли второй аргумент (вернётся NaN)
calculateDiscount(100);

// Проблема 2: Передали не тот тип (вернётся NaN)
calculateDiscount("100$", 20);

// Проблема 3: Перепутали местами аргументы
calculateDiscount(20, 100); 
```
В JS мы узнаем об этих ошибках только когда код упадет в браузере у пользователя. Вот почему нам нужен TypeScript — чтобы ловить это прямо в редакторе!

## 4. 📖 Теория

Чтобы типизировать функцию в TypeScript, нам нужно описать две вещи:
1. **Параметры (что входит)** — типизируются так же, как обычные переменные.
2. **Тип возвращаемого значения (что выходит, ReturnType)** — пишется после закрывающей скобки параметров через двоеточие.

**Синтаксис (Type Signature / Сигнатура типа):**
```typescript
function имяФункции(параметр1: Тип1, параметр2: Тип2): ТипВозврата {
  // код
}
```

**Если функция ничего не возвращает:**
Иногда функции нужны просто чтобы что-то сделать (вывести в консоль, изменить внешний стейт), но они не делают `return`. В JS такие функции неявно возвращают `undefined`. В TypeScript для этого есть специальный тип `void` (пустота).

*Связь с JS:* В рантайме вся эта типизация исчезнет, останется обычный JS код. Типизация функций — это секьюрити-контроль на входе в клуб твоей функции.

## 5. 💡 Примеры с детальными комментариями

### Пример 1: Простая функция (hello world уровень)
```typescript
// Принимает строку, ничего не возвращает (void)
function greetUser(name: string): void {
  console.log(`Привет, ${name}!`);
  // return здесь вызовет ошибку TS, так как мы указали void
}

greetUser("Алекс"); // Работает отлично
// greetUser(42);   // Ошибка TS: аргумент типа 'number' нельзя назначить параметру типа 'string'
```

### Пример 2: Чуть сложнее (возвращает значение)
```typescript
// Принимает два числа, возвращает число
function multiply(x: number, y: number): number {
  return x * y; // Если попытаться вернуть строку, TS выдаст ошибку!
}

const result = multiply(5, 10); // TS понимает, что result — это number
```

### Пример 3: Реальный use case (типы-объекты в аргументах)
Часто мы передаем в функцию целый объект. Лучше всего описать его через `type` (Type Alias, который мы прошли в Уроке 8!).
```typescript
type User = {
  id: string;
  name: string;
  age: number;
}

// Функция принимает объект типа User и возвращает boolean
function isAdult(user: User): boolean {
  return user.age >= 18;
}

const boy: User = { id: "1", name: "Влад", age: 16 };
const canBuyAlcohol = isAdult(boy); // false
```

## 6. 🔍 JS vs TS сравнение

**Было в JS:**
```javascript
const formatPrice = (price, currency) => {
  return `${price} ${currency}`;
};
formatPrice(100); // Вернет "100 undefined" (ошибка по невнимательности)
```

**Стало в TS:**
```typescript
const formatPrice = (price: number, currency: string): string => {
  return `${price} ${currency}`;
};
// formatPrice(100); // TS не даст собрать код: "Expected 2 arguments, but got 1."
```
*Преимущество:* Ошибки ловятся во время написания кода. Редактор сам подскакивает и говорит: "Эй, ты забыл валюту передать!". Разве не чудо?

## 9. 🛠️ Практический совет
Всегда старайся **явно** указывать возвращаемый тип функции (`: Type { ... }`), даже если TypeScript может догадаться (type inference) сам по `return`.
Почему? Если внутри сложной функции ты случайно сделаешь `return` не того типа данных, TS без явного указания типа просто скажет "ну ладно, теперь функция возвращает ЭТО". Явное указание — это твоя страховка от самого себя.

## 📝 Дополнительные заметки

**Вопрос:** Почему при создании двух файлов (`practice.ts` и `solution.ts`) с одинаковыми именами функций/типов TypeScript ругается на "Duplicate function implementation / Duplicate identifier"?
**Ответ:** По умолчанию TypeScript считает все файлы в проекте, в которых НЕТ `import` или `export`, **глобальными скриптами** (они все находятся в одной глобальной области видимости). Поэтому TS видит два `Product` в одном проекте и бьет тревогу. 
Чтобы решить это, достаточно добавить `export {};` в конец файла. Это превращает файл в изолированный **модуль**, и имена перестают конфликтовать!
