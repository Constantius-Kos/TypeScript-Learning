# Заметки — Урок 8: Never и Unknown типы

## ❓ Вопросы и ответы из сессии

### Q: `any` — это как чистый JS, а `unknown` — как union всех типов с обязательной проверкой?

**`any` = чистый JS** — да, точно. Ты говоришь TS: "заткнись и не проверяй". Это как выключить TypeScript для конкретной переменной.

**`unknown` похоже на union всех типов, но есть разница:**

С обычным union TS **знает** что внутри:
```ts
let a: string | number = getValue();
a.toString(); // ✅ — этот метод есть у обоих типов
```

С `unknown` TS **вообще не знает** что внутри — поэтому запрещает всё:
```ts
let u: unknown = getValue();
u.toString(); // ❌ — TS: "понятия не имею что это, не дам"
```

`unknown` — как union всех типов **без общих методов**. Ничего нельзя пока не проверишь.

Если одной фразой:
- `any` — *"я знаю что делаю, не мешай"*
- `unknown` — *"я не знаю что это, заставь меня проверить"*

### Q: Почему в catch error имеет тип unknown если мы его не указывали?

Это встроенное поведение TypeScript начиная с версии 4.0.

До TS 4.0 `error` был `any` — TS молчал и разрешал всё. Разработчики поняли что это дыра: бросить можно что угодно, а TS делал вид что там всегда `Error`.

С TS 4.0 изменили на `unknown` — чтобы заставить разработчика проверить тип перед использованием. Это специальное правило только для `catch` — нигде больше TS не ставит `unknown` автоматически.

Без проверки разрешено только:
```ts
catch (error) {
    console.log(error); // ✅ console.log принимает any
    throw error;        // ✅ перебросить можно
    error.message;      // ❌ обращение к свойствам — нельзя
}
```

### Q: `unknown` используется везде где возможны разные типы?

Нет — для "разных вариантов типов" есть union:
```ts
// Знаешь возможные типы — используй union
function process(value: string | number) { ... }
```

`unknown` используется конкретно когда **вообще не знаешь что придёт** — данные из внешнего мира:
```ts
JSON.parse(str)   // что внутри? неизвестно
fetch(url)        // что вернёт API? неизвестно
catch (error)     // что бросили? неизвестно
```

Везде где `unknown` — обязательно проверяй тип перед использованием.

**Если одной фразой: union = знаешь варианты, unknown = не знаешь ничего.**

### Q: Почему при проверке `typeof data === "object"` нужно дополнительно проверять `data !== null`?

Потому что `typeof null === "object"` — исторический баг JavaScript. `null` хранился как объект в памяти, и это нельзя исправить из-за обратной совместимости.

Поэтому проверяют в два шага:
```ts
typeof data === "object"  // отсеивает string, number, boolean... но null пропускает!
data !== null             // явно отсеиваем null
```

Без второй проверки `null` прошёл бы дальше и на `"users" in data` упало бы с ошибкой.

### Q: Если null это "object" через typeof — почему он falsy?

Это две разные независимые системы JS:
- `typeof` — про то как значение **хранится** в памяти
- Булевость — про то является ли значение **"чем-то истинным"**

```ts
typeof null        // "object"  ← как хранится (баг JS)
Boolean(null)      // false     ← является ли "чем-то"
null instanceof Object // false ← реальный объект? нет
```

Аналогично с пустой строкой:
```ts
typeof ""    // "string" ← тип: строка
Boolean("")  // false    ← истинное? нет, пустая
```

**Falsy значения в JS** (ведут себя как false при проверке):
```ts
Boolean("")        // false
Boolean(0)         // false
Boolean(null)      // false
Boolean(undefined) // false
Boolean(NaN)       // false
Boolean(false)     // false
```

### Q: Можно ли заменить цепочку проверок на `??`?

Нет — это разные операторы с разными задачами:

```ts
// ?? — выбирает между двумя значениями (замена тернарника)
const name = user.name ?? "Аноним";
// если null/undefined — берём "Аноним", иначе — берём user.name

// && — цепочка проверок, все должны быть истинны
typeof data === "object" && data !== null && "users" in data
// каждая проверка должна пройти, иначе останавливаемся
```

- `??` — *"что использовать если значения нет"*
- `&&` — *"все условия должны выполниться"*

### Q: В switch проверяется свойство kind, тогда почему в assertNever передаётся весь объект?

Разница между тем что **проверяет switch** и что **отслеживает TS**:

- `switch` проверяет конкретное свойство `shape.kind`
- TS отслеживает весь объект `shape`

Аналогия: посылки на складе. Смотришь на **ярлык** (kind) чтобы понять что внутри. Но когда отправляешь дальше — отправляешь **всю посылку**, не только ярлык.

```ts
switch (shape.kind) {       // смотрим на ярлык
    case "circle":          // ярлык "circle" → обработали всю посылку
    case "square":          // ярлык "square" → обработали всю посылку
    default:
        assertNever(shape)  // отправляем всю необработанную посылку
}
```

TS думает: *"circle и square уже обработаны — значит в default осталась только посылка с ярлыком triangle"*. Поэтому `shape` в default сужается до `{ kind: "triangle", base: number, height: number }`.

### Q: Почему при возврате `parseResult` после длинной цепочки проверок TS ругается и нужен `as`?

Логика верная, но TS после цепочки `&&` с `in` оператором сужает тип только до:
```ts
object & Record<"name", unknown> & Record<"age", unknown>
```

Он знает что поля существуют, но **забывает** что ты уже проверил их типы через `typeof`. Это ограничение алгоритма narrowing в TS — он не всегда отслеживает все шаги в длинной цепочке.

Решение — `as`, это стандартный подход в таких ситуациях:
```ts
return parseResult as { name: string; age: number }
```

`as` не обход системы типов — это способ сказать: *"я вручную проверил всё что нужно, результат вот такой"*. Именно для таких случаев он и существует.
