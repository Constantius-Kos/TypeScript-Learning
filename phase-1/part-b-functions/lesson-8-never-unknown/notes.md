# Нотатки — Урок 8: Never і Unknown типи

## ❓ Питання і відповіді з сесії

### Q: `any` — це як чистий JS, а `unknown` — як union всіх типів з обов'язковою перевіркою?

**`any` = чистий JS** — так, точно. Ти кажеш TS: "заткнись і не перевіряй". Це як вимкнути TypeScript для конкретної змінної.

**`unknown` схоже на union всіх типів, але є різниця:**

Зі звичайним union TS **знає** що всередині:
```ts
let a: string | number = getValue();
a.toString(); // ✅ — цей метод є в обох типів
```

З `unknown` TS **взагалі не знає** що всередині — тому забороняє все:
```ts
let u: unknown = getValue();
u.toString(); // ❌ — TS: "гадки не маю що це, не дам"
```

`unknown` — як union всіх типів **без спільних методів**. Нічого не можна поки не перевіриш.

Якщо однією фразою:
- `any` — *"я знаю що роблю, не заважай"*
- `unknown` — *"я не знаю що це, змуси мене перевірити"*

---

### Q: Чому в catch error має тип unknown якщо ми його не вказували?

Це вбудована поведінка TypeScript починаючи з версії 4.0.

До TS 4.0 `error` був `any` — TS мовчав і дозволяв все. Розробники зрозуміли що це діра: кинути можна що завгодно, а TS робив вигляд що там завжди `Error`.

З TS 4.0 змінили на `unknown` — щоб змусити розробника перевірити тип перед використанням. Це спеціальне правило тільки для `catch` — ніде більше TS не ставить `unknown` автоматично.

Без перевірки дозволено тільки:
```ts
catch (error) {
    console.log(error); // ✅ console.log приймає any
    throw error;        // ✅ перекинути можна
    error.message;      // ❌ звернення до властивостей — не можна
}
```

### Q: `unknown` використовується скрізь де можливі різні типи?

Ні — для "різних варіантів типів" є union:
```ts
// Знаєш можливі типи — використовуй union
function process(value: string | number) { ... }
```

`unknown` використовується конкретно коли **взагалі не знаєш що прийде** — дані із зовнішнього світу:
```ts
JSON.parse(str)   // що всередині? невідомо
fetch(url)        // що поверне API? невідомо
catch (error)     // що кинули? невідомо
```

Скрізь де `unknown` — обов'язково перевіряй тип перед використанням.

**Якщо однією фразою: union = знаєш варіанти, unknown = не знаєш нічого.**

### Q: Чому при перевірці `typeof data === "object"` потрібно додатково перевіряти `data !== null`?

Тому що `typeof null === "object"` — історичний баг JavaScript. `null` зберігався як об'єкт в пам'яті, і це не можна виправити через зворотну сумісність.

Тому перевіряють у два кроки:
```ts
typeof data === "object"  // відсіює string, number, boolean... але null пропускає!
data !== null             // явно відсіюємо null
```

Без другої перевірки `null` пройшов би далі і на `"users" in data` впав би з помилкою.

### Q: Якщо null це "object" через typeof — чому він falsy?

Це дві різні незалежні системи JS:
- `typeof` — про те як значення **зберігається** в пам'яті
- Булевість — про те чи є значення **"чимось істинним"**

```ts
typeof null        // "object"  ← як зберігається (баг JS)
Boolean(null)      // false     ← чи є "чимось"
null instanceof Object // false ← реальний об'єкт? ні
```

Аналогічно з порожнім рядком:
```ts
typeof ""    // "string" ← тип: рядок
Boolean("")  // false    ← істинне? ні, порожнє
```

**Falsy значення в JS** (поводяться як false при перевірці):
```ts
Boolean("")        // false
Boolean(0)         // false
Boolean(null)      // false
Boolean(undefined) // false
Boolean(NaN)       // false
Boolean(false)     // false
```

### Q: Чи можна замінити ланцюжок перевірок на `??`?

Ні — це різні оператори з різними завданнями:

```ts
// ?? — вибирає між двома значеннями (заміна тернарника)
const name = user.name ?? "Анонім";
// якщо null/undefined — беремо "Анонім", інакше — беремо user.name

// && — ланцюжок перевірок, всі повинні бути істинними
typeof data === "object" && data !== null && "users" in data
// кожна перевірка повинна пройти, інакше зупиняємося
```

- `??` — *"що використовувати якщо значення немає"*
- `&&` — *"всі умови повинні виконатися"*

### Q: В switch перевіряється властивість kind, тоді чому в assertNever передається весь об'єкт?

Різниця між тим що **перевіряє switch** і що **відстежує TS**:

- `switch` перевіряє конкретну властивість `shape.kind`
- TS відстежує весь об'єкт `shape`

Аналогія: посилки на складі. Дивишся на **ярлик** (kind) щоб зрозуміти що всередині. Але коли відправляєш далі — відправляєш **всю посилку**, не тільки ярлик.

```ts
switch (shape.kind) {       // дивимося на ярлик
    case "circle":          // ярлик "circle" → обробили всю посилку
    case "square":          // ярлик "square" → обробили всю посилку
    default:
        assertNever(shape)  // відправляємо всю необроблену посилку
}
```

TS думає: *"circle і square вже оброблені — значить в default залишилася тільки посилка з ярликом triangle"*. Тому `shape` в default звужується до `{ kind: "triangle", base: number, height: number }`.

### Q: Чому при поверненні `parseResult` після довгого ланцюжка перевірок TS лається і потрібен `as`?

Логіка вірна, але TS після ланцюжка `&&` з оператором `in` звужує тип тільки до:
```ts
object & Record<"name", unknown> & Record<"age", unknown>
```

Він знає що поля існують, але **забуває** що ти вже перевірив їх типи через `typeof`. Це обмеження алгоритму narrowing в TS — він не завжди відстежує всі кроки в довгому ланцюжку.

Рішення — `as`, це стандартний підхід в таких ситуаціях:
```ts
return parseResult as { name: string; age: number }
```

`as` — це не обхід системи типів — це спосіб сказати: *"я вручну перевірив все що потрібно, результат ось такий"*. Саме для таких випадків він і існує.
